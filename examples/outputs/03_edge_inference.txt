# ANALYSIS

## 1. TTFT Constraint Assessment

**The 10ms TTFT target is not achievable with any cloud GPU option for Llama 3.1 7B.** Here's why:

- **Typical TTFT for 7B models on H100/H200** (FP8, vLLM/TensorRT-LLM, optimized kernels):
  - Short prompts (32-128 tokens): ~15-30ms
  - Medium prompts (256-512 tokens): ~30-60ms
- **Physical limits:**
  - Network RTT EU→provider DC: 5-20ms (AWS eu-west-1, Azure West Europe, GCP europe-west4)
  - Kernel dispatch + prefill compute: 8-25ms for 7B FP8
  - Framework overhead (Python GIL, scheduler): 2-5ms

**Closest achievable: ~18-25ms TTFT** with:
- H200 (4.8 TB/s memory bandwidth → faster prefill)
- Single-GPU deployment (no multi-GPU dispatch overhead)
- Optimized TensorRT-LLM FP8 kernel
- Co-located in EU (minimize network RTT)

**Interpretation:** I will treat this as a **latency-sensitive edge deployment** requirement and recommend the **lowest-latency cloud option** while explicitly flagging the ~10ms gap.

For **low concurrency edge**, the cost structure favors:
- **Single-GPU deployment** (no multi-instance overhead)
- **On-demand or spot pricing** (utilization likely <30%, reserved commitment doesn't amortize)
- **Smallest instance size** to minimize idle cost

**Low concurrency assumption:** If the user is serving <10 req/sec, utilization is likely <10-20%. This makes **on-demand pricing with autoscaling** the optimal choice, not reserved.
